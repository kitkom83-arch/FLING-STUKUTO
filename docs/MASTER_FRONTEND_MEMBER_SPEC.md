# Master Frontend Member Spec

Phase AJ status: docs/static member frontend specification only.

Safety boundary: no production DB, no real money, no live provider/payment/bank/SMS/Slip OCR, no migration, no deploy, and no runtime write action is enabled by this document.

## Status Legend

| Status | Meaning |
| --- | --- |
| mock implemented | Existing or planned screen can run with mock/static data only. |
| read-only implemented | Existing API can safely display data in local/staging. |
| static spec only | UX/API contract is documented, but no current runtime integration is confirmed. |
| future guarded write | Write action requires auth, validation, idempotency where needed, audit, and safe environment gates. |
| future live integration | Requires certification before any live provider/payment/bank/SMS/Slip OCR use. |

## Cross-Screen Requirements

- Member JWT and admin JWT must stay separate.
- Member routes must return only the logged-in member's data.
- All money/reward-like writes need backend validation and idempotency before broader staging rollout.
- Frontend must never decide wallet balance, payout status, provider transfer result, or Lucky Wheel reward result.
- Response handling must avoid raw token, password, secret, database URL, auth header, stack trace, raw user-agent, and unmasked IP output.
